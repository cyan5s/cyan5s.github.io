信息泄露漏洞学习1

Created2025-10-16|Updated2025-11-22|WEB
|Post Views:

因设计、配置或代码缺陷，无意中向用户（尤其是未授权用户）暴露了敏感信息

原理

一、服务器与中间件泄露

主要由错误的配置或默认行为引起

  1. 默认错误页面与调试信息

    • 场景： 应用程序在生产环境中启用了调试模式。当出现错误（如数据库连接失败、代码语法错误）时，服务器不是向用户返回一个友好的错误页面，而是返回包含*堆栈跟踪（Stack Trace） 的详细错误信息。*
    • 泄露内容： 绝对服务器路径、数据库连接字符串、SQL查询语句片段、代码片段、服务器使用的框架和库的版本。
    • 案例： 访问 https://example.com/user.php?id=abc（传入一个非数字参数），页面返回 Warning: mysql_fetch_array() expects parameter 1 to be resource, boolean given in /var/www/html/app/database.php on line 55。这直接泄露了路径和数据库类型。

    查找方法：

    • 触发错误： 在URL参数、表单字段、HTTP头（如 Cookie, User-Agent）中提交异常数据：特殊字符：' " ; -- /*（触发SQL或XSS错误）超长字符串：AAAAAAAA...[数万个A]...AAA（触发缓冲区溢出或处理异常）错误数据类型：例如向需要数字的参数 id=1提交 id=abc。
    • 工具辅助： 使用Burp Suite的 Active Scan 功能或 Nuclei 模板

  2. 目录列表（Directory Listing）

    • 场景： Web服务器（如Apache、Nginx、IIS）的某个目录下没有默认首页文件（如 index.html, index.php），且服务器配置为允许显示目录内容。
    • 泄露内容： 该目录下的所有文件名和子目录结构。攻击者可以像在文件管理器中一样浏览，直接发现备份文件、配置文件、上传的敏感文件等。
    • 案例： 访问 https://example.com/assets/，浏览器直接显示一个包含 backup_20231027.zip, config.ini.bak, uploads/的列表。

    查找方法：

    • 目录枚举： 使用目录爆破工具，并对所有发现的目录进行访问，观察响应。
      • 工具： dirsearch, gobuster, ffuf。
      • 命令示例： dirsearch -u https://target.com -e php,html,zip,bak
    • 常见路径： 手动尝试访问常见路径，如 /uploads, /assets, /images, /admin, /backup, /tmp

  3. 冗余与备份文件

    • 场景： 开发人员在线上环境遗留了用于开发、测试或备份的文件。
    • 泄露内容：
      • 备份文件： .bak, .old, .txt, .zip, .tar.gz等后缀的文件可能包含源代码或数据库备份。
      • 版本控制文件： .git目录、.svn目录、.DS_Store（macOS系统文件）。如果能被直接访问，攻击者可以利用工具（如 githacker）下载完整的源代码仓库。
      • 编辑器临时文件： .swp(Vim), .swo, .~(某些IDE) 文件可能包含正在编辑的未保存内容。
    • 案例： 著名的 .git泄露事件屡见不鲜，攻击者下载源码后分析，发现了数据库密码硬编码在配置文件中。

    查找方法：

    • 文件枚举： 使用字典对目标进行文件爆破，字典应包含备份文件常见后缀。
      • 常用后缀： .bak, .old, .swp, .swo, .tar.gz, .zip, .7z, .rar, .txt, .src, .dist.
      • 特殊文件： /.git/HEAD, /.DS_Store, /.env, /web.config, /phpinfo.php。
    • 工具： 同上（dirsearch, gobuster），但使用专门的文件字典。
    • Git泄露利用： 发现 /.git/目录后，使用 GitHacker 或 dvcs-ripper 工具尝试下载完整源码库。

    验证： 访问文件URL，返回200状态码并可下载或查看文件内容

  4. HTTP响应头信息

    • 场景： Web服务器、应用程序框架或编程语言默认会在HTTP响应头中添加标识自身的字段。
    • 泄露内容： 服务器软件和精确版本（Server: nginx/1.18.0）、后端编程语言和版本（X-Powered-By: PHP/7.4.33）、使用的框架（X-Generator: Django 3.2）。

    查找方法：

    • 手动分析： 使用浏览器开发者工具（F12）的“网络(Network)”面板，或使用 curl命令，检查所有HTTP请求的响应头。
      • 命令示例： curl -I https://target.com（-I选项只获取头部）
    • 工具辅助： Burp Suite 的 Proxy 或 Repeater 模块可以方便地查看和对比所有响应头。

    关键字段： 检查 Server, X-Powered-By, X-AspNet-Version, X-Generator, X-Debug-Token等字段的值。

二、应用程序逻辑泄露

应用程序自身的设计和代码逻辑缺陷。

  1. 用户输入导致的错误泄露

    • 场景： 用户通过参数（如URL参数、表单字段、HTTP头）提交了应用程序未预料到的数据，导致程序异常，并将错误细节返回。
    • 泄露内容： 同“调试信息”，但由用户输入触发。
    • 案例： 在搜索框输入一个单引号 '，页面返回SQL语法错误，暴露了数据库类型和部分表结构。

    查找方法：

    • 模糊测试（Fuzzing）： 对所有用户输入点进行系统性的异常输入测试。
      • 输入点： GET/POST参数、Cookie、JSON请求体、XML文件上传、HTTP头。
      • 工具： Burp Suite的 Intruder 或 Scanner 模块是首选。也可以使用 ffuf。
    • 步骤： 提交测试Payload -> 捕获响应 -> 筛选出与其他响应不同的报文（不同状态码、不同长度、包含错误关键词）。

    验证： 响应中包含系统抛出的详细错误信息，而非统一的错误页面

  2. 用户与数据枚举

    • 场景： 应用程序在不同操作（如登录、注册、密码重置）返回的差异信息，允许攻击者判断某些信息是否存在。
    • 泄露内容： 有效的用户名、邮箱地址、手机号。
    • 案例：
      • 登录功能： 输入正确用户名错误密码提示“密码错误”，输入错误用户名则提示“用户不存在”。
      • 注册功能： 输入已存在的邮箱提示“该邮箱已注册”。

    查找方法：

    • 对比分析： 对登录、注册、密码重置、用户名检查等功能进行测试。
      • 测试用例：
        • 登录：使用有效用户+错误密码 vs 无效用户+任意密码。
        • 注册：使用已存在的邮箱 vs 未存在的邮箱。
        • 密码重置：输入已知的用户名/邮箱 vs 未知的。
    • 对比内容：
      • HTTP状态码： 是否不同（如200 vs 404）。
      • 响应体内容： 错误信息是否不同（如“密码错误” vs “用户不存在”）。
      • 响应时间： 检查用户是否存在是否导致响应时间有显著差异（时间盲注）。
      • 响应长度： 两个响应的字节长度是否不同。
    • 工具： Burp Suite的 Comparer 工具是进行差异对比的神器。

    验证： 通过差异可以可靠地推断出某个数据（如用户名）是否存在。

  3. API响应过度

    • 场景： API接口在响应请求时，返回了远多于前端所需的数据对象，通常是数据库记录的完整映射。
    • 泄露内容： 用户密码哈希、内部用户ID、其他用户的隐私数据、系统内部信息。
    • 案例： 调用 GET /api/users/me获取当前用户信息，API却返回了 {id: 123, name: "alice", email: "alice@example.com", password_hash: "...", is_admin: true, ...}，包含了敏感字段。

    查找方法：

    • 权限切换测试： 使用不同权限的用户（尤其是低权限用户）访问同一个API端点，观察返回的数据结构。
      • 步骤：
        • 用普通用户A登录，访问 GET /api/users/123。
        • 用普通用户B登录，访问 GET /api/users/123。
        • 用管理员登录，访问同一个端点。
        • 对比三个响应，看低权限用户是否能获取到本不应看到的数据（如他人的邮箱、私人信息）。
    • 工具： Burp Suite的 Repeater 模块，配合不同身份认证的Cookie或Token，方便地重放和对比请求。

    验证： 响应中包含了当前用户权限范围之外的敏感数据字段。

三、云与第三方服务配置错误

现代应用依赖大量外部服务，配置不当会导致严重泄露。

  1. 云存储桶公开访问

    • 场景： 将阿里云OSS、AWS S3、Google Cloud Storage等云存储服务设置为“公开可读”（Public Read），导致任何知道链接的人都可以下载文件。
    • 泄露内容： 用户上传的私人文件、应用程序源码、配置文件、数据库备份。
    • 案例： 数亿条用户数据因S3桶配置错误而泄露的新闻屡见报端。

    查找方法：

    • 被动侦查：
      • 搜索引擎语法： 在Google等搜索引擎中搜索 site:s3.amazonaws.com <company_name>或 site:oss-cn-hangzhou.aliyuncs.com <company_name>。
    • 主动扫描：
      • 工具扫描： 使用专门针对云存储桶的工具，它们通过字典来猜测桶名并检查其权限。
        • AWS S3： s3scanner, bucket-stream, awscli（aws s3 ls s3://bucket-name/）
        • 阿里云 OSS： ossutils。
      • 步骤： 工具通常会生成可能的桶名（如 company-prod, company-assets, company-backup）并尝试列出其内容。

    验证： 能够匿名地列出存储桶内的文件列表，或直接通过URL下载到文件。

  2. 公开的监控与调试端口

    • 场景： 将用于系统监控、调试的服务（如Redis, Memcached, MongoDB, Elasticsearch）暴露在公网，且未设置认证。
    • 泄露内容： 整个数据库的内容、服务器性能指标、内部网络信息。

    查找方法：

    • 端口扫描： 使用 nmap 或 masscan 对目标IP段进行常用端口扫描。
      • 关键端口： 6379(Redis), 27017(MongoDB), 9200(Elasticsearch), 11211(Memcached), 5601(Kibana)。
      • 命令示例： nmap -p 6379,27017,9200 -sV target-ip-range
    • 空间测绘引擎： 使用 Shodan, Censys, FoFa 直接搜索开放了这些服务的IP。
      • 搜索语法：
        • port:6379 redis
        • product:mongodb
        • "elasticsearch" "server: Elasticsearch"
    • 匿名连接测试： 扫描发现端口开放后，使用客户端工具尝试匿名连接。Redis示例： redis-cli -h <target-ip>连接后尝试执行 INFO命令。

    验证： 无需任何认证即可连接到服务并执行查询或管理命令。

总结一下：

  • 大范围，进行适当的漏扫

  • 范围小，手动注入时：

    • 明白每种漏洞可能存在的位置，利用该漏洞可能存在的方法进行测试（输入回显）

之后根据例子学习每个漏洞（【0】是SQL注入 目录扫描 弱口令）
